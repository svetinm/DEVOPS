LO1 Ś Use of containers and container services
Q1: Run the httpd container in the background, post port 80 to host port 8081, give it a name and hostname.
Answer:
Use podman run -d -p 8081:80 --name myweb --hostname webhost docker.io/library/httpd. Confirm with podman inspect myweb and check the 'Name' and 'Hostname' fields.

Q2: Run the container with --restart=always, kill its main process, and prove that Podman restarts it.
Answer:
The container with --restart=always automatically restarts when its process ends. Use podman kill to send SIGKILL, then check podman ps and RestartCount in inspect output.
podman run -d --restart=always --name restartme docker.io/library/nginx
podman kill restartme
sleep 2 && podman ps
podman inspect restartme | grep RestartCount

Q3: Run the container with --memory=256m --cpus=0.5 and check the limits.
Answer:
Memory and CPU limits are set at startup. podman stats displays live usage, and podman inspect displays HostConfig.Memory and NanoCpus values.
podman run -d --memory=256m --cpus=0.5 --name limited nginx
podman stats limited --no-stream
podman inspect limited | grep -E 'Memory|NanoCpus'

Q4: Run a container with --env KEY=VALUE and another with --env-file, show the difference.
Answer:
--env KEY=VALUE sets a single variable inline, while --env-file reads variables from a file (format KEY=VALUE, one in order). Both methods are seen in 'podman exec <c> env'.
echo 'DB_HOST=localhost
DB_PORT=5432' > myenv.txt
podman run -d --env APP=inline --name c1 nginx
podman run -d --env-file myenv.txt --name c2 nginx
podman exec c1 env | grep APP
podman exec c2 env | grep DB_

Q5: Create a user-defined bridge network, run a container on it, and find the IP address.
Answer:
Podman user-defined networks support DNS resolution by container name. It uses podman network create to create, --network flag at startup, and podman network inspect to view IPs.
podman network create mynet
Podman Run-D --Network Minet --Webnode NGINX
podman network inspect mynet | grep -A5 webnode

Q6: Pause and pause the container. What is the state of the processes during the break?
Answer:
podman pause sends a SIGSTOP to all processes within the container Ś the processes are frozen in memory, do not receive CPU time, but do not lose state. subman unpause sends a SIGCONT and continues execution.
A pause <name>
podman ps  # STATUS: Paused
Unpause <name>

Q7: Rename an existing container with a subname rename.
Answer:
Podman rename changes the name of the container without rebooting. Useful when the name needs to be aligned with conventions without losing data or container states.
podman rename oldname newname
Podman PS --Filter Name=NewName

Q8: Use podman logs with --tail, --since, and -f flags.
Answer:
--tail N shows the last N lines of logs. --since 10m shows logs from the last 10 minutes. -f (follow) tracks the logs in real time as 'tail -f'.
podman logs --tail 20 <c>

podman logs -f <c>

Q9: Extract a single field (e.g. IP address) using podman inspect --format with Go template syntax.

Q10: Copy the file from the host to the container and back with subman cp.
Answer:
The CP is working in a way. The format is 'source:destination' where a prefix with the name of the container and a colon indicates the path within the container.
echo 'test' > host.txt

Q11: Execute an interactive shell in a container, install a package Ś does this change survive the container recreation?
Answer:
Changes within the container (installed packages, modified files) do NOT survive subman rm and reboot from the same image. A container filesystem is ephemeral unless volume or subman commit is used.
podman exec -it <c> /bin/bash
# apt install curl -y
podman commit <c> myimage:modified # if you want to save

Q12: Use podman top and podman stats to show processes and resource usage.
Answer:
The Podman Top displays the processes inside the container (such as PS aux). podman stats displays live CPU, memory, network I/O, and block I/O for running containers.
podman top <c>
podman stats <c> --no-stream

Q13: Run the container with --rm. When is auto-removal useful and what do you lose?
Answer:
--rm automatically deletes the container when it stops. Useful for one-time tasks (debug, tests, scripts) as it prevents the accumulation of stalled containers. You lose logs and the ability to inspect after stopping.
podman run --rm --name temp alpine echo 'done'
# The container no longer exists
podman ps | grep temp # empty

Q14: Bind-mount the directory in the container as read-only and prove that the writes were rejected.
Answer:
Adding :ro at the end of the mount path prevents writes from the container to the host directory. An attempt to write returns a "Read-only file system" error.
mkdir /tmp/hostdir && echo 'readonly' > /tmp/hostdir/file.txt
podman run --rm -v /tmp/hostdir:/data:ro alpine sh -c 'echo test > /data/new.txt'
# Error: Read-only file system

podman ps --filter label=env=prod

Q16: Commit the modified container to a new image with the subman commit.

Q17: Show filesystem changes within a container with subman diff. What do A/C/D tags mean?
Answer:
Podman diff compares the current filesystem container with the original image. A=Added (new file), C=Changed (modified file), D=Deleted (deleted file).
podman diff <c>

Q18: Run the container with --network none. Demonstrate network unavailability and provide a use case.
Answer:
--network none completely isolates the container from the network. There's no loopback (except for the interface). Useful for security-sensitive workloads, offline processing, or build containers that are not allowed to access the Internet.
podman run --rm --network none alpine ping -c1 8.8.8.8
# ping: network not available

Q19: Publish port only to a specific host IP address (-p 127.0.0.1:8082:80).
Answer:
By binding the port to 127.0.0.1 (localhost), the port is only available locally on the host, not from other IP addresses. Useful when you don't want to expose the service to external networks.

Q20: Use the subman port to list the published ports of the container.

The subman port shows the port mapping of the container. The same data is also available in subman inspect under HostConfig.PortBindings and NetworkSettings.Ports.

Q21: Run the container as a non-root user with --user.
Answer:
--user can receive a UID, GID, or username. The process within the container executes with this identity, which is a security good practice for reducing privileges.
podman run --rm --user 1000:1000 alpine id
podman run --rm --user nobody alpine whoami

Q22: Generate a systemd unit for the container and explain how to run it at boot.
Answer:
Podman Generate Systemd creates a .service file. Place it in ~/.config/systemd/user/ and enable it with systemctl --user enable. Alternatively, the Quadlet .container files in ~/.config/containers/systemd/ automatically generate a unit at systemd daemon-reload.
podman generate systemd --new --name <c> > ~/.config/systemd/user/myapp.service
systemctl --user daemon-reload
systemctl --user enable --now myapp.service

Q23: Run podman events in one terminal while starting/stopping a container in another.
Answer:
Podman Events streams lifecycle events in real time: create, start, stop, remove, exec, etc. Each event has a timestamp, type, container name, and action. Useful for debugging and auditing.
# Terminal 1:
podman events
# Terminal 2:
podman run --rm --name test alpine echo hi
# You will see: create, start, died, remove


LO2 Ś Managing and Creating Container Images
Q1: Write a Containerfile that uses ARG to select the base at build.
Answer:
ARG declares a variable that can be passed at build with --build-arg. It can be used in the FROM statement for parameterized base selection. Valid only for the build phase.
FROM alpine:${VERSION}
ARG VERSION=3.20
RUN echo built with version ${VERSION}
 
# Build:
podman build --build-arg VERSION=3.19 -t test:3.19 .
podman build --build-arg VERSION=3.20 -t test:3.20 .

Q2: Write a multi-stage Containerfile that copies only the result to the final minimal image.
Answer:
A multi-stage build uses multiple FROM statements. The first phase (builder) compiles/builds the artifact. The second phase copies only the necessary artifact with COPY --from=builder. The result is a far smaller final image because it does not contain build tools.
FROM golang:1.22 AS builder
WORKDIR /src
COPY . .
RUN go build -o app .
 
FROM alpine:3.20
COPY --from=builder /src/app /app
CMD ["/app"]

Q3: Add a .containerignore file and demonstrate excluding files from the build context.
Answer:
.containerignore (analogous to .dockerignore) defines file/directory patterns that are not sent to the build context. It reduces the size of the context and prevents sensitive files from being sent (e.g. . Secrets, and secrets.
#.containerignore:
*.log
.env
node_modules/
.git/
 
podman build . -t myapp
# Files from .containerignore will not be available in COPY commands

Q4: Build an image and apply three tags at once. Explain the registry/namespace/name:tag convention.
Answer:
Convention: registry.io/namespace/name:tag Ś e.g. docker.io/library/nginx:1.25. Registry is the host, namespace is the user/organization, name is the name of the image, tag is the version. 'latest' is the default tag, but it's not always the latest!
podman build -t myapp:1.0 -t myapp:1 -t myapp:latest .
podman images | grep myapp

Q5: Look at the history layer of the picture with the sub-history. How to reduce the number of layers?
Answer:
Each RUN, COPY, ADD command creates a new layer. To reduce layers, combine the RUN commands with && and clear the cache in the same layer. Less layer = smaller image and faster pulling.
podman history nginx:latest
# Optimization: combine RUN commands
# RUN apt update && apt install -y curl && rm -rf /var/lib/apt/lists/*

Q6: Write a Containerfile that sets the non-root USER and WORKDIR.
Answer:
The USER command changes the user for all subsequent RUN, CMD, ENTRYPOINT commands. The user must exist in the image (created with useradd/adduser in the previous RUN). WORKDIR creates and sets up a working directory.
FROM ubuntu:24.04
RUN useradd -m appuser
USER appuser
WORKDIR /home/appuser/app
CMD ["whoami"]
 
# Test:
podman run --rm myimage  # ispisuje: appuser

Q7: Add HEALTHCHECK to the Containerfile and display the health status.
Answer:
HEALTHCHECK defines a command by which Podman periodically checks the health of containers. Status can be: starting, healthy, unhealthy. Podman ps displays the status in the HEALTH column.
HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD curl -f http://localhost:80/ || exit 1
 
podman ps --format '{{. Names}} {{. Status}}'
podman healthcheck run <c>

Q8: Demonstrate the difference between ENTRYPOINT and CMD in Containerfile.
Answer:
ENTRYPOINT is a fixed command that is always executed. CMD are the default arguments that can be overridden at startup. Together: ENTRYPOINT is 'executable', CMDs are 'default args'. CMD is overrun at podman run, ENTRYPOINT is not (except with the --entrypoint flag).
FROM alpine
ENTRYPOINT ["echo"]
CMD ["Hello World"]
 
podman run myimage            # echo Hello World
podman run myimage Hello # echo Hello (run over by cmd)

Q9: Explain the difference between ADD and COPY. Write a Containerfile that justifies the use of each.
Answer:
COPY simply copies files from the host to the image Ś recommended for all normal cases. ADD has additional capabilities: it automatically unpacks .tar archives and can fetchat URLs. Use ADD only when you need auto-extract; otherwise always COPY.
FROM ubuntu:24.04
COPY app.py /app/app.py # obi?ni copy
ADD archive.tar.gz /opt/data/     # auto-extract
ADD https://example.com/cert.pem /etc/ssl/  # URL fetch

Q10: Save the image to tarball with podman save, delete it and restore it with podman load.
Answer:
podman save creates OCI-compatible tarball images. Useful for air-gapped transfers (without internet), image backup, or image sharing without registry.
The Man Who Saved nginx.tar docker.io/library/nginx:latest
podman rmi nginx:latest
Podman Load-E NGINX.
podman images | grep nginx

Q11: Drag the image by digest (name@sha256:...). Why is it more reproducible than a tag?
Answer:
A tag (e.g. :latest) can be reassigned to another image Ś the same tag does not guarantee the same content. The digest (sha256 hash of the manifest) is cryptographically unique for the exact content of the image and never changes.
podman pull nginx@sha256:abc123...
# Always the same exact image, regardless of what happens to the tag

Q12: Build an image with --no-cache. Explain layer caching and when disabling it is useful.
Answer:
Podman caches layers by default Ś if the command and context have not changed, the previously built layer is used. --no-cache forces a rebuild of all layers. Useful when: you need the latest packages (apt update), debug caching problems, or CI/CD must guarantee a fresh build.
podman build --no-cache -t myapp:fresh .
# vs normal build that uses cache:
podman build -t myapp:cached .

Q13: Build from the non-default Containerfile name using -f.
Answer:
-f specifies the path to the Containerfile/Dockerfile. The build context (dot at the end) is the directory from which the relative COPY paths are calculated. Useful for a higher-level build organization with multiple Containerfile variants.
podman build -f Containerfile.prod -t myapp:prod .
podman build -f docker/Containerfile.dev -t myapp:dev ./docker

Q14: Use COPY --chown to set the ownership of the file in the image.
Answer:
--chown on the COPY command sets the UID:GID ownership of the copied files directly when copying, without the additional RUN chown layer. It's more efficient and doesn't add unnecessary layer.
FROM ubuntu:24.04
RUN useradd -m appuser
COPY --chown=appuser:appuser app/ /home/appuser/app/
RUN ls -la /home/appuser/app/ # vlasnik i appuser

Q15: Write a Containerfile combining ARG and ENV. Demonstrate the difference (build-time vs run-time).
Answer:
ARG only exists during build; It's not available in the running container. ENV exists in both the image and the runtime container (visible in the 'env' command). ARG can be used to initialize ENV values.
ARG BUILD_DATE
ENV APP_VERSION=1.0
ENV BUILT_ON=${BUILD_DATE} # ARG -> ENV
 
podman build --build-arg BUILD_DATE=$(date) -t myapp .
podman run --rm myapp env | grep -E 'APP_VERSION|BUILT_ON'

Q16: Log in and push the image to a private repository. Where are credentials stored?
Answer:
Podman login stores credentials in ~/.config/containers/auth.json (rootless) or /run/containers/0/auth.json. The format is base64-encoded JSON with authtokens by registry. The file is used automatically for all pull/push operations.
podman login quay.io
podman tag myapp:latest quay.io/username/myapp:latest
podman push quay.io/username/myapp:latest
cat ~/.config/containers/auth.json

Q17: Remove dangling images from the subman image prune. What does 'dangling' mean?
Answer:
A dangling image is an image without a tag (displayed as <none>:<none>). They are created when the same image is retagged or rebuilt Ś the old version is left without reference. Podman Image Prune deletes all dangling images. Deletes all unused images.
podman images | grep '<none>'  # dangling slike
podman image prune             # briÜe dangling
podman image prune -a # deletes all unused
podman rmi specific-image:tag # deletes a specific image

Q18: Write a Containerfile for a static file server using python -m http.server.
Answer:
Python -m http.server serves files from the current directory. EXPOSE documents the port (but does not open it Ś this is what -p does at startup). WORKDIR must be a directory with files.
FROM python:3.11-slim
WORKDIR /srv/web
COPY static/ .
EXPOSE 8000
CMD ["python", "-m", "http.server", "8000"]

Q19: Build an image that installs a specific pinana version of the package. Why pinning important?
Answer:
Pinning the version (e.g. nginx=1.24.0) ensures reproducibility: the same build always installs the same package, with no unexpected changes due to new versions. Without pinning, rebuild can intrude breaking changes.
FROM ubuntu:24.04
RUN apt-get update && \
    apt-get install -y nginx=1.24.0-2ubuntu7 && \
    rm -rf /var/lib/apt/lists/*

Q20: Scans an unknown image with podman image inspect to detect Entrypoint, Cmd, ports, and env.
Answer:
podman image inspect returns the complete metadata of the image including the Config object with Entrypoint, Cmd, ExposedPorts, and Env. This tells how the image works without running or reading the Containerfile.
podman image inspect redis:7 | python3 -m json.tool | grep -A5 '"Config"'

Q21: Write a Containerfile with a multi-line RUN that installs packages and cleans the cache in the same layer.
Answer:
Each run creates a layer. If you clear the cache in a special RUN, the previous layer already contains the cache and it cannot be reduced. The delete must be in the SAME RUN to reduce the layer.
# CORRECT - cleaning in the same layer:
RUN apt-get update && \
    apt-get install -y curl git && \
    rm -rf /var/lib/apt/lists/*
 
# WRONG - cache remains in the previous layer:
RUN apt-get update && apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*

Q22: Tag and push the same images in two different registries (Docker Hub and Quay). When is image mirroring useful?
Answer:
Image mirroring distributes the image across multiple registries for: high availability (one registry unavailable), disaster recovery, latency reduction (geographically closer to the registry), avoiding rate limits (Docker Hub free tier).
podman tag myapp:latest docker.io/username/myapp:latest
podman tag myapp:latest quay.io/username/myapp:latest
podman push docker.io/username/myapp:latest
podman push quay.io/username/myapp:latest

Answer:
The minimal base image (alpine, distroless, slim) has a smaller attack surface (fewer potential vulnerabilities), smaller pull and push sizes, faster launch. Fewer packets = less CVE risk.

podman run --rm ubuntu dpkg -l | toilet -l

podman run --rm alpine apk info | wc -l

LO3 Ś Application Delivery, Network Architecture, and Component Security
Q1: Create a pod with subman under create, add app and DB container, prove communication via localhost.
Answer:
Containers within the same floor share a network namespace Ś they communicate through localhost because they have the same network stack. It's like processes on the same machine. The port must be published on the floor, not on individual containers.
Podman Term Create --MyPod --P 8080:80
Podman Run-D --Term MyPod --Namesake Webap NGINX
Podman Run -D --Term MyPod --Named DB Postgrace
# webapp can access DB at localhost:5432

Q2: Create a user-defined network, prove DNS resolution by container name.
Answer:
There is no DNS resolution on the default Podman network. On user-defined networks, Podman embeds DNS that resolves container names to their IP addresses. Containers can be called by name instead of IP address.
podman network create appnet
podman run -d --network appnet --name db postgres:15
podman run -d --network appnet --name app mywebapp
podman exec app ping -c2 db  # rjeÜava 'db' -> IP

Q3: Deploy a two-tier stack (e.g. Gitea + PostgreSQL) and complete the first boot.
Answer:
A two-tier stack consists of an application layer and a database. The key is: the database is started first, the app waits for the DB to be ready, the data is persisted in volume, the app is configured to connect to the DB by name within the same network.
podman network create gitea-net
podman volume create gitea-db-data
Podman Run-D--Network Gitea-Net --Named Giteya-DB \
  -e POSTGRES_PASSWORD=secret -e POSTGRES_DB=gitea \
  -v gitea-db-data:/var/lib/postgresql/data postgres:15
 
podman run -d --network gitea-net -p 3000:3000 \
  -e GITEA__database__DB_TYPE=postgres \
  -e GITEA__database__HOST=gitea-db:5432 gitea/gitea

Q4: Persist the database data in the named volume. Prove that the data survives container recreation.
Answer:
Named volume is Podman's managed storage that exists independently of the container. When a container is deleted and re-created with the same volume, the data is intact.
podman volume create dbdata
podman run -d --name db -v dbdata:/var/lib/postgresql/data postgres
# Create a table, add data
Podman RM -F DB
# Restart with the same volume:
podman run -d --name db2 -v dbdata:/var/lib/postgresql/data postgres
# The data is still there!

Q5: Use podman secret to pass the database password instead of --env.
Answer:
Podman Secret Create keeps sensitive data encrypted. --secret flag mounts secret as a file in /run/secrets/<name>. More secure than --env because the password does not appear in the subman inspect output, env process, or shell history.
echo 'supersecret' | podman secret create db_password -
podman run -d --secret db_password \
  -e MYSQL_ROOT_PASSWORD_FILE=/run/secrets/db_password \

Podman Secret LS

Q6: Write a podman compose file (compose.yaml) for the app + database, run with podman compose up -d.
Answer:
compose.yaml defines a multi-container application declaratively. The Services section defines containers, networks, network, volumes, and persistent storage. podman compose up -d runs everything in the background.

    image: nginx
    ports: ['80:80']
    depends_on: [db]
  db:
    image: postgres:15
    environment:
      POSTGRES_PASSWORD: secret
    Volumes: [pgdata:/var/lib/postgresql/data]
volumes:
  pgdata:
 
podman compose up -d
podman compose ps

Q7: In the compose file, it keeps the database on the internal network with no published port, expose only the app.
Answer:
By defining internal and external networks, the database may be inaccessible from the host but accessible to the application. The base connects only to the internal network, the app connects to both or only to the external with access to the internal one.
networks:
  frontend:
  backend:
    internal: true # No access from the host
services:
  app:
    ports: ['80:80']
    networks: [frontend, backend]
  db:
    networks: [backend]  # samo backend, nema published porta

Q8: Add compose healthcheck and depends_on: condition: service_healthy for the boot order.
Answer:
Without a healthcheck, depends_on just waiting for the container to start (not necessarily ready). S condition: service_healthy, the app waits until DB healthcheck becomes 'healthy'. Prevents connection errors during initialization.
services:
  db:
    image: postgres
    healthcheck:
      test: ["CMD", "pg_isready", "-U", "postgres"]
      interval: 10s
      retries: 5
  app:
    depends_on:
      db:
        condition: service_healthy

Q9: Use env_file: in the compose file for database credentials instead of inline values.
Answer:
env_file: Loads variables from a file instead of being hardcoded in compose.yaml. The file is not committed to Git (add to .gitignore). Format: KEY=VALUE, one in order. It is more secure than the inline environment value in the compose file.
# .env.db:
POSTGRES_PASSWORD=secret
POSTGRES_USER=app
POSTGRES_DB=mydb
 
# compose.yaml:
services:
  db:
    image: postgres:15
    env_file: .env.db

Q10: Scale the compose service to multiple replicas with --scale and explain the distribution of requests.
Answer:
podman compose up --scale app=3 creates 3 instances of the app service. Load balancing between replicas depends on the configuration Ś Compose itself does not balance HTTP, a reverse proxy (nginx) is required in front. Port binding does not work with replicas without a proxy.
podman compose up -d --scale app=3
podman compose ps  # prikazuje 3 app instance

Q11: Mount bind mount for configuration and named volume for data in the same container. Explain the difference.
Answer:
Bind mount (-v /host/path:/container/path) directly mounts the host directory Ś good for configuration (easy to edit from the host, changes visible immediately). Named volume (-v volname:/path) is Podman's managed storage Ś good for data (no dependency on the host path, transferable between hosts).
podman run -d \
  -v ./config/app.conf:/etc/app/app.conf:ro \
  -v appdata:/var/lib/app/data \
  myapp

Q12: Backup the named volume in the tarball using the helper container, and restore to the new volume.
Answer:
Standard technique: run a small container (alpine/busybox) that mounts both the source volume and bind mount directories on the host, and then tars the content. Restoration is the opposite.
# Backup:
podman run --rm -v appdata:/data:ro -v $(pwd):/alpine backup \
  takes czf /backup/appdata.tar.gz -C /data .
 
# Restore:
podman volume create appdata-new
podman run --rm -v appdata-new:/data -v $(pwd):/backup alpine \
  takes xzf /backup/appdata.tar.gz -C /data

Q13: Place the nginx reverse proxy in front of the app container on the shared network.

nginx in proxy mode forwards requests to an upstream server by name (DNS) within Podman's network. The guest host nginx port is 80, nginx forwards it to the app:port within the internal network.
# nginx.conf:
server {
  listen 80;
  location / {
    proxy_pass http://app:3000;
  }
}

  -v ./nginx.conf:/etc/nginx/conf.d/default.conf:ro nginx

Q6: Set up a RollingUpdate strategy with maxSurge:1 and maxUnavailable:0. What does this mean for availability?
Answer:

Q8: Add CPU/memory requests and limits to Deployment and make sure they are running under spec.

spec:

LO5 Ś Troubleshooting Container Application Delivery
Q1: ERROR: podman run -d --name web -p 80:8080 nginx (nginx listens at 80, port not available).
Answer:
The port mapping format is -p HOST:CONTAINER. Error: Host port is 80, container port is 8080, but nginx listens at 80 in container. Fixed: -p 80:80 (or -p 8080:80 if you want to access 8080 on the host).
# WRONG:
podman run -d -p 80:8080 nginx # nginx ne sluÜa na 8080
# CORRECT:
podman run -d -p 80:80 nginx # nginx sluÜa na 80
# or:
podman run -d -p 8080:80 nginx # available on host:8080

Q2: GREŖKA: podman run -d --name db -e MYSQL_ROOT_PASSWORD docker.io/library/mysql (kontejner odmah izlazi).
Answer:
Problem: --env KEY without =VALUE does not set a variable with a value Ś the shell would have to have that variable. MySQL requires MYSQL_ROOT_PASSWORD with value. Check with podman logs db.
# WRONG:
podman run -d -e MYSQL_ROOT_PASSWORD mysql
# CORRECT:
podman run -d -e MYSQL_ROOT_PASSWORD=tajnaLozinka mysql
# or:
podman run -d -e MYSQL_ALLOW_EMPTY_PASSWORD=true mysql

Q3: ERROR: podman run -d --network host -p 8080:80 nginx (-p flag is ignored).
Answer:
S --network host container shares host network namespace Ś no container network, no port mapping. All port publish (-p) flags are ignored. Nginx listens directly to host port 80. Access localhost:80, not localhost:8080.
# -p is ignored with --network host
# nginx directly listens to host:80
curl localhost:80# works
curl localhost:8080 # not working
# If you need 8080, configure nginx to listen on 8080

Q4: ERROR: podman run -d --name c1 busybox (immediately Exited(0)). Why and how to repair?
Answer:
Busybox Image doesn't have a default long-running command Ś it comes out immediately. A container lives as long as its main process. Run with a command that doesn't end: sleep infinity, or interactively with -it.
# WRONG:
podman run -d busybox # comes out immediately
# CORRECT:
podman run -d busybox sleep infinity # remains running
# or interactively:
podman run -it busybox sh

Q5: ERROR: podman run --rm -d alpine echo hello, so podman logs job doesn't work.
Answer:
--rm deletes the container as soon as it stops. Detached (-d) with a short-lived command (echo) means: container start, execute echo, finish, delete immediately. There is no container for which logs would work. Solution: don't combine --rm with -d for short tasks, or use only -d without --rm.
# --rm + -d + short command = container disappears before log read
# SOLUTION - without --rm:
podman run -d --name job alpine echo hello
podman logs job
podman rm job # manually delete

Q6: GREŖKA: FROM debian:12 / RUN apt-get install -y nginx (build fails).
Answer:
The Debian apt cache is not available immediately Ś you need apt-get update before installing. Without an update, apt does not know about available packages and reports errors.
# WRONG:
RUN apt-get install -y nginx
# CORRECT:
RUN apt-get update && \
    apt-get install -y nginx && \
    rm -rf /var/lib/apt/lists/*

Q7: GREŖKA: FROM python:3.11 / CMD python app.py (signal handling problemi).
Answer:
The shell form (CMD python app.py) runs the process as /bin/sh -c 'python app.py'. SIGTERM is sent to the shell process, not to the python process Ś the app does not receive a signal for graceful shutdown. The exec form (JSON array) runs directly on Python as PID 1.
# POGREŖNO (shell form):
CMD python app.py
# CORRECT (exec form):
CMD ["python", "app.py"]
# Python then receives the SIGTERM directly

Q8: ERROR: FROM node:20 / COPY . /app / RUN npm install (each code change reinstalls npm).
Answer:
Docker/Podman invalidates the cache from the changed line onwards. COPY . copies all Ś each change to the app code invalidates the cache for npm install. Solution: First, copy only package.json and install the dependencies, then copy the rest of the code.
# WRONG (npm install repeats every time):
COPY . /app
RUN npm install
 
# CORRECT (npm install is repeated only when the package.json is changed):
COPY package.json package-lock.json /app/
WORKDIR /app
RUN npm install
COPY . /app

Q9: ERROR: FROM ubuntu:24.04 / EXPOSE 8080 / CMD python3 -m http.server 3000 (nothing responds to 8080).
Answer:
EXPOSE only documents which port the container uses Ś it doesn't open the port automatically. We also notice a mismatch: EXPOSE 8080 but the server is listening at 3000. Fix: Synchronize EXPOSE and server port.
# WRONG:
EXPOSE 8080
CMD ["python3", "-m", "http.server", "3000"]
# CORRECT:
EXPOSE 3000
CMD ["python3", "-m", "http.server", "3000"]

LO6 Ś Evaluation of container orchestration systems (theoretical)
Q1: Analyze the Kubernetes network model compared to the Docker network.
Answer:
Kubernetes network: each pod gets its own IP (flat network), the floors communicate directly without NAT (the CNI plugin ensures this). Docker: containers are behind a bridge network with NAT outward, port publish is explicit. K8s is more complex but scalable Ś there are no limitations to port publishing, the Service abstracts load balancing.

Q2: Compare Kubernetes storage abstraction (CSI, StorageClasses) with Podman's storage plugins. Which is more flexible for stateful workloads?
Answer:
Kubernetes is more flexible for production stateful workloads. CSI (Container Storage Interface) standardizes the interface for storage providers (AWS EBS, GCE PD, NFS, Ceph). StorageClasses enable dynamic provisioning Ś PVC automatically creates storage. StatefulSet + PVC per-replica guarantees isolated storage. Podman offers simple volume drivers that are sufficient for local development, but not for multi-node production.

Q3: Evaluate the pattern operator (CRDs + controllers) for managing stateful services compared to manual Kubernetes manifests.
Answer:
Operators encapsulate domain-specific knowledge (e.g., how to upgrade PostgreSQL without losing data) into a controller loop that automates complex operations. Advantages: automatic failover, backup, rolling upgrades, self-healing. Disadvantages: complexity, debugging is more difficult. Manual manifests give full control, but require a lot of operational knowledge and manual intervention.

Q4: Evaluate the difference in developer experience between Plain Kubernetes and OpenShift S2I.
Answer:
Plain Kubernetes optimizes for flexibility and control Ś the developer must understand and create Docker/Podman images, write manifests, manage the registry. OpenShift S2I (Source-to-Image) optimizes for delivery speed Ś the developer smokes the code, the platform automatically detects the runtime, builds the image, and deploys. S2I reduces the barrier to entry but hides infrastructure details.

Q5: Compare Docker and Podman in terms of architecture and rootless security.
Answer:
Docker uses the daemon architecture Ś dockerd works as a root service, each docker command communicates with the daemon. Security risk: compromised Docker socket = root access. Podman is daemonless Ś each podman command is a fork-exec with no central process. Rootless Podman runs containers as a user without privileges, using user namespaces. For security-conscious teams, Podman is more secure because it doesn't have an attack surface central daemon process.

Q6: Defend the small team recommendation: Docker Compose on a single host vs Kubernetes.
Answer:
For a small team with 2-5 services and predictable traffic, Compose on a single host is a rational choice: zero operational complexity, fast deployment, easy debugging, minimal costs. Kubernetes makes sense when: you need auto-scaling, multi-node HA, advanced rolling update, IaC with a large ecosystem. Switching to k8s is an investment Ś scaling your infrastructure too early is an anti-pattern.

Q7: Evaluate the migration of workloads from OpenShift to Kubernetes.
Answer:
OpenShift adds abstractions on top of Kubernetes (Routes instead of Ingress, BuildConfig, DeploymentConfig, ImageStreams, SCCs). Migrating to vanilla Kubernetes requires: replacing Routes with Ingress + cert-manager, BuildConfigs with CI/CD pipeline (Tekton/GitLab CI), DeploymentConfigs with standard Deployments. Security policies (SCCs) map to PodSecurityAdmission. Migration is technically feasible but operationally demanding.

Q8: Critically evaluate OpenShift vs vanilla Kubernetes security. Why does the enterprise prefer it in regulated industries?
Answer:
OpenShift has a stricter default security profile: Security Context Constraints (SCCs) prohibit root by default, all pods must use a non-root UID. It has built-in: audit logging, compliance profiles (CIS), OAuth integration, image signing. Vanilla Kubernetes is more permissive by default Ś it requires manual security configuration. For banks, healthcare, government institutions: Red Hat provides commercial support, certifications (FedRAMP, HIPAA), long-term lifecycle support.

Q9: Would you choose Kubernetes for enterprise-grade applications? Argument.
Answer:
Yes, Kubernetes is the de facto standard for enterprise container workload. Arguments: massive CNCF ecosystem (Helm, Prometheus, Istio, Argo), vendor-neutrality prevents lock-in, horizontal scaling to thousands of nodes, declarative API with GitOps capabilities, multi-cloud portability. Matuting the platform through 10+ years, a large community, rich documentation. Cons: high learning curve, operational complexity of self-managed clusters Ś managed service (EKS, GKE, AKS) is solved.

Q10: Would you choose OpenShift for enterprise-grade applications? Argument.
Answer:
Yes, OpenShift is justified for an enterprise that requires an integrated, vendor-supported package. Advantages: Red Hat commercial support with SLA, built-in CI/CD (Tekton/Pipelines), developer console, operator hub, security hardening defaults. Less operational overhead than self-managed Kubernetes. Cons: higher license cost, vendor lock-in, slower adoption of Kubernetes novelties. Ideal for regulated industries or teams without deep Kubernetes expertise that need enterprise support.
